# ngbot Codebase Map

> Originally generated by Cartographer and reconciled with the implementation on 2026-07-14 after the durability refactor. 119 Go files, ~30.5k Go LOC, 33 migrations, 30 challenge locales, 4131-line i18n.

## System Overview

**ngbot** is a Telegram chat gatekeeper bot that protects groups from spam and automated bot joins. It provides CAPTCHA-based entry verification (inline buttons **and** a join-request WebApp Mini App with TLS-terminated fallback), LLM-powered spam detection with Gemini explicit caching, external-quote heuristic fast-path, provenance-aware multi-provider ban checking (lols.bot + CAS), durable community voting moderation, MarkdownV2 greeting templates with entity-based normalization, reaction profile checks, and daily statistics tracking.

## Observed behavior and invariants

| Surface | Current invariant |
|---|---|
| Update processing | `main/buildRuntime` supplies the ordered `admin → gatekeeper → reactor` handler slice. A keyed dispatcher preserves FIFO within a chat, processes different chats in parallel, caps active workers at 8, and applies backpressure at the Telegram buffer budget. |
| Settings | Cached settings have value semantics: clone on ingress/egress, pure timeout getters, publish only after successful persistence, and warmup fills only missing keys. |
| Gatekeeper | Every challenge generation has a unique `challenge_id`. State transitions use expected-state CAS; Telegram effects are retried by a lifecycle worker, and durable rows are deleted only after the external action succeeds or is already applied. Activation failures compensate before returning. After 8 failures the row and last error remain durable but leave automatic polling for operator intervention. |
| Moderation | Spam cases persist `resolve_at` and resolving states. Vote and timeout race through one transactional claim; restart recovery continues both due and partially resolved cases, with exactly one stats increment. Telegram resolution retries stop after 8 failures while retaining the case and error. |
| Voting eligibility | A voter must be a member or administrator of the source chat. Log-channel subscription alone grants no moderation authority. If the log post fails, an in-chat surface must exist and its binding must be persisted before destructive moderation starts. |
| Banlist | `banlist_sources` stores provider/feed provenance and expiry. A successful provider snapshot replaces only that provider; a failed provider retains its prior snapshot. `banlist` remains the compatibility projection of all unexpired sources. |
| SQLite | Every connection enables foreign keys; startup verifies the pragma, performs a ping, and closes the handle on error. Confirmed legacy orphans are removed before enforcement. |
| Security | Secrets are fully redacted, classification logs contain IDs/length/digest rather than message text, diagnostic commands are authorized, and public WebApp URLs require HTTPS except loopback development. |

**Known limitations**: SQLite still has one process-wide `RWMutex`; it should be narrowed only after the concurrent-read/ban-import benchmark demonstrates material contention. Telegram idempotency is inferred from known already-applied responses; an exhausted retry remains durable and is logged for operator intervention. The WebApp listener remains plain HTTP inside the container and therefore requires a TLS-terminating reverse proxy.

## Module Guide

### Core Bot Module (`internal/bot/`)

**Purpose**: Central bot service, custom polling, and update processing pipeline

| File | Purpose | Key Exports |
|------|---------|-------------|
| `service.go` | Membership cache and immutable settings snapshots | `NewService()`, `IsMember()`, `GetSettings()`, `SetSettings()`, `warmupCache()` |
| `update_processor.go` | Ordered chain-of-responsibility dispatch | `UpdateProcessor`, `MessageType` constants, `ExtractContentFromMessage` |
| `update_dispatcher.go` | Keyed scheduling, concurrency cap, backpressure, panic isolation | `KeyedDispatcher`, `NewKeyedDispatcher()`, `Submit()`, `Stop()` |
| `telegram.go` | Context-aware Telegram API calls | `Send()`, `GetChat()`, `GetChatMember()`, `RequestBool()` |
| `polling.go` | Custom long-polling with exponential backoff + recovery window | `GetUpdatesChans()`, `PollingOptions`, `PollingRecoveryError` |
| `dependencies.go` | Interface contracts | Narrow `Service` (settings/membership/language) and `Handler` |

**Key Patterns**:
- Two-level caching: members (5-min TTL with lazy eviction), settings (process-lifetime immutable snapshots, no TTL)
- `cmd/ngbot/buildRuntime` is the composition root and supplies the configured handler slice; there is no global registry or global config singleton
- Chain of responsibility remains `admin → gatekeeper → reactor`; `proceed bool` stops the chain
- Update timeout: 5 minutes (stale updates dropped); `IsMember` re-verifies via Telegram `GetChatMember` on cache miss
- Custom polling: exponential backoff (initial 1s, max 30s), recovery window (default 10m), malformed-update filtering (`isStructurallyEmptyUpdate`)
- Keyed dispatcher: FIFO per chat, concurrent progress across chats, user/global fallback key, at most 8 active workers, pending budget equal to `BotAPI.Buffer`, blocking backpressure, and per-update panic recovery
- Shutdown stops intake, cancels in-flight I/O, and drains workers within the lifecycle deadline

**Dependencies**: `telegram-bot-api`, `db`, `config`, `i18n`, `errgroup`

### Lifecycle Module (`internal/lifecycle/`)

**Purpose**: Component lifecycle orchestration with ordered start/stop

**Key Patterns**:
- `Component` interface: `Start(ctx) error`, `Stop(ctx) error`
- Start in registration order; **rollback** on first failure and return `errors.Join(startError, rollbackErrors)`
- Stop in reverse (LIFO) order; all errors collected via `errors.Join`

**Registered components** (in order): `bot.service`, `BanService`, `SpamControl`, `Gatekeeper` (owns the WebApp HTTP server), `Admin`, `updateLoopComponent` (polling).

**Access gating (3 layers)**:
1. `/settings` must run **in-group** first (proves caller is a manager + bot is a member); anonymous-admin messages (`msg.SenderChat != nil`) are deleted and ignored.
2. `/start settings_` private deep-link **re-verifies** manager status via `GetChatMember`.
3. Every callback + free-text input re-runs `ensureManagerAccess` — a demoted admin is locked out mid-session.

**Admin Panel Session Lifecycle**:
1. User runs `/settings` in group → deep-link button → private chat
2. Session created with JSON state, 1-hour TTL
3. Each callback → `findPanelSessionCommand` (brute-force split resolution) → `applyPanelCommand` → re-render
4. Background goroutine cleans expired sessions every 5 minutes
5. Buttons carry DB-backed `AdminPanelCommand` rows (callback_data = `<base64 sessionID>_<base64 commandID>`) to bypass Telegram's 64-byte limit; old commands deleted at each render
6. `saveChatSettings` called on every settings mutation → also marks recommended protection as handled in KV

**Three join paths**:

1. **Direct/public join** — `chat_member`/`new_chat_members` → restrict → persist challenge generation → send and bind the inline keyboard. Activation failures compensate the restriction/message instead of exposing an unsolvable CAPTCHA. Correct answers claim `unrestrict_pending`; wrong×3/expiry atomically claims `reject_pending`.

2. **Join request via DM** (no WebApp, or no QueryID) — `chat_join_request` persists the query binding before external work, then creates a private challenge (`CommChatID`=userDM). Correct → `approve_member_pending`; the worker approves and transitions to `passed_waiting_member_join`. The later `chat_member` event sends the greeting and deletes that exact generation.

3. **Join request via WebApp** (QueryID present, `PublicURL` configured) — the row stores `ChallengeID`, `WebAppToken`, `JoinRequestQueryID`, and options before Telegram presentation. A valid correct POST atomically claims `approve_query_pending`; the worker applies `AnswerJoinRequestQuery(approve)` and transitions to `passed_waiting_member_join`.

**WebApp fallback-on-timeout**: if the WebApp is never opened (`web_app_opened_at IS NULL` for >11s, swept every 3s) or expires while pending, `BeginDMFallback` atomically flips the same generation to `web_app_fallback_pending`. The query binding remains durable while the worker prepares, sends, and binds the DM challenge. A restart continues whichever step is incomplete.

**Challenge Status Lifecycle**:

```mermaid
stateDiagram-v2
    [*] --> pending: generation persisted and presentation bound
    pending --> web_app_fallback_pending: BeginDMFallback
    web_app_fallback_pending --> pending: DM message attached
    pending --> approve_query_pending: correct WebApp answer
    pending --> approve_member_pending: correct DM join answer
    pending --> unrestrict_pending: correct public answer
    pending --> reject_pending: max attempts or expiry
    approve_query_pending --> passed_waiting_member_join: query approved
    approve_member_pending --> passed_waiting_member_join: member approved
    unrestrict_pending --> [*]: member unrestricted and row deleted
    reject_pending --> [*]: ban/decline applied and row deleted
    passed_waiting_member_join --> [*]: join observed or TTL cleanup

    note right of approve_query_pending
        External-action states carry next_attempt_at,
        attempt_count, and last_error.
        Restart recovery resumes them.
    end note
```

**Status Constants** (7): `pending`, `web_app_fallback_pending`, `approve_query_pending`, `approve_member_pending`, `unrestrict_pending`, `reject_pending`, `passed_waiting_member_join`. Terminal completion is row deletion only after the required Telegram effect succeeds or is recognized as already applied; expiry is computed from `expires_at`.

**Gatekeeper Constants**:
- `captchaSize = 5`, `maxChallengeAttempts = 3`
- `approvedJoinRequestChallengeTTL = 5 * time.Minute`
- `webAppOpenDeadline = 11 * time.Second` (unopened-WebApp fallback threshold)
- `joinCaptchaInitDataTTL = 1 * time.Hour` (init-data replay window)
- Allowed options counts: `{3, 4, 5, 6, 8, 10}`
- Process intervals: new joiners = 1m, expired challenges = 1m, **unopened WebApp = 3s**, due durable actions = lifecycle worker sweep
- Greeting placeholders: `{user}`, `{chat_title}`, `{chat_link_titled}`, `{timeout}`
- Greeting template prefix: `mdv2:` tags MarkdownV2 templates; absence = legacy Markdown

**WebApp security hardening** (in `gatekeeper_webapp.go`):
- **init-data HMAC validation**: `api.ValidateWebAppData(bot.Token, initDataRaw)`; then `parseWebAppInitData` extracts `query_id`, `user.id`, `auth_date`.
- **Replay protection**: `auth_date` rejected if older than 1h (`joinCaptchaInitDataTTL`).
- **Binding**: `initData.UserID == challenge.UserID` and (non-test) `initData.QueryID == challenge.JoinRequestQueryID`, else 403.
- **Generation-bound approval claim**: `ClaimForApproval(challengeID)` — atomic `UPDATE … WHERE challenge_id=? AND status='pending'`; `claimed==false` → 409. Stale requests cannot mutate a replacement challenge.
- **Durable approval**: a Telegram failure leaves `approve_query_pending` with retry metadata. It never rolls back to `pending`; the lifecycle worker resumes after restart and deletes no state before the side effect completes.
- **Known-banned re-check**: `banChecker.IsKnownBanned(userID)` re-checked **after** a correct answer but **before** claiming (N2 fix).
- **Cross-site guard**: `isJoinCaptchaCrossSiteMutation` blocks non-GET/HEAD whose `Sec-Fetch-Site: cross-site` or whose `Origin` host ≠ `Host`.
- **Body cap**: `http.MaxBytesReader` at 16 KiB. **Crawler blocklist** (gptbot, claudebot, …) → 403.
- **CSP**: nonce-based `script-src`/`style-src`, `connect-src 'self'`, `frame-ancestors 'none'`. Answer text is XOR-obfuscated in HTML (real gate is the HMAC, not the obfuscation).
- **HTTP server timeouts** (slowloris hardening — N10 fix): `ReadHeaderTimeout` 5s, `Read`/`Write` 15s, `Idle` 60s.
- **Routes**: `/robots.txt` (deny-all), `/sitemap.xml` (empty), `/gatekeeper/join-captcha` (GET), `/gatekeeper/join-captcha/answer` (POST).

**Pipeline Stages** (in order — short-circuit, each records `result.Stage`/`SkipReason`):

```
Message arrives
    │
    ▼
[1] trustedSenderChat? (group itself / linked channel / auto-forward) → YES → SKIP
    │
    NO
    ▼
[2] IsMember? → YES → SKIP
    │
    NO
    ▼
[3] isChatAdministrator? (live GetChatMember) → YES → SKIP
    │
    NO
    ▼
[4] IsChatNotSpammer? (manual override) → YES → remember + SKIP
    │
    NO
    ▼
[5] banService.CheckBan → BANNED → ProcessBannedMessage (delete + ban, no voting)
    │
    NOT BANNED
    ▼
[6] IsChatKnownNonMember? → YES → handleKnownNonMember (re-verify, may SKIP)
    │
    NO
    ▼
[7] LLMFirstMessageEnabled? → NO → remember + SKIP
    │
    YES
    ▼
[8] Extract content → EMPTY → SKIP
    │
    HAS CONTENT
    ▼
[9] detectFirstMessageExternalQuoteHeuristic → TRIGGERED → immediate spam + StatHeuristicSpam
    │
    NOT TRIGGERED
    ▼
[10] normalizeCyrillics → load per-chat examples (max 20) → LLM IsSpam + StatLLMChecked
         │
         ├── SPAM → processSpam (voting ON) or processBanned (voting OFF) + rememberAuthorIfPossible
         └── NOT SPAM → rememberAuthorIfPossible (promote to member / remember non-member)
```

> **Ordering note**: members and admins (stages 2–3) never reach the banlist. The known-non-member check (6) comes **after** the ban check (5), so a banned user who is also a remembered non-member is treated as banned first.

**Reaction Profile Check** (`reactor_reaction_profile_check.go`, gated by `settings.ReactionProfileCheckEnabled`): when an unknown user or anonymous channel **reacts** (additions only; removals ignored), builds a text profile (full name, @username, personal-chat bio/pinned/posts up to 10) and runs it through the LLM. Spam → wipe reactions + ban (permanent for actor-chats). Membership is double-checked live before escalating. This catches spammers who only drop a reaction and never text the group. Has an admin-panel toggle (ReactionProfileCheck leaf screen).

**`voteBanCommand`** (report flow): triggered by `/voteban` (replying to a suspect) **or** mentioning the bot in a reply. Sequence: (1) report-specific LLM check → confirmed spam = instant ban; (2) reporter has restrict permission = instant ban; (3) community voting disabled = refuse; (4) otherwise `processReported` starts a message-bound community vote.

**Community Voting Policy Resolution**:

| Step | Description |
|------|-------------|
| 1. Base policy | From `config.SpamControl` global config |
| 2. Per-chat overrides | From `db.Settings` (if not `SettingsOverrideInherit` = -1) |
| 3. Normalization | Timeout min 5m, MinVoters min 1, MaxVoters min 0, MinVoters% min 0 |
| 4. Required voters | `max(MinVoters, len(members) * MinVoters% / 100)`, capped by MaxVoters, min 1; member-list failure defers resolution instead of substituting zero |

**Voting Resolution Logic**:

| Scenario | Result |
|----------|--------|
| Spam majority | `spam` status → user banned + `StatSpamConfirmed` |
| Not-spam majority | `false_positive` status → user unmuted + `StatFalsePositive` |
| Tie + timeout | `false_positive` (user-friendly) |
| Insufficient votes + timeout | `false_positive` (user-friendly) |

Only members/administrators of the **source chat** can vote. `AddVoteIfPending` and `ClaimResolution` serialize vote/timeout races in SQLite. The winner first enters `resolving_spam` or `resolving_false_positive`; the worker applies the Telegram effect, then `FinalizeResolution` writes the terminal state and increments the matching stat once in the same transaction. On startup, due `pending` and unfinished `resolving_*` cases are recovered. There are no process-local per-case timers.

**Voting presentation and retention**: a log-channel post is preferred, but failure creates an in-chat voting surface before the original message is deleted or the suspect is muted. The prompt remains usable through resolution; the chat prompt is deleted afterward and the log-channel keyboard is removed. `/voteban`/mention report messages form a durable cleanup queue: rows older than 10 minutes are deleted from Telegram first and from SQLite only after that idempotent effect succeeds.

### LLM Adapters (`internal/adapters/llm/`)

**Purpose**: Abstraction over OpenAI and Google Gemini APIs

| Adapter | Default Model | Key Features |
|---------|---------------|--------------|
| `gemini.API` | `gemini-2.5-flash-lite` | Explicit context caching (6h TTL), local fingerprint→handle cache, `BLOCK_NONE` safety, **empty-cache retry** |
| `openai.API` | `gpt-4o-mini` | Custom base URL; cache-unaware (`Cacheable` ignored); logger discarded |

**Shared Config**: Temperature 0, TopP 1, MaxTokens 4 (binary spam classification output)

**Gemini Explicit Caching Architecture**:

```mermaid
flowchart TD
    A[ChatCompletion called] --> B[splitPromptSegments]
    B --> C{Has cacheable prefix?}
    C -->|No| D[Full uncached request]
    C -->|Yes| E[loadOrCreateCache]
    E --> F{Unexpired local handle?}
    F -->|Yes| G[Use cached prefix + live tail]
    F -->|No| H[List remote cache once, or create with 6h TTL]
    H --> G
    G --> I{Result?}
    I -->|Text returned| J[Return response]
    I -->|Empty response OR cache-use error| K[Fallback: full uncached request]
    K --> J
```

**Cacheable Message Contract** (`ChatCompletionMessage.Cacheable bool`): system prompt + built-in few-shot examples (~50) → `Cacheable: true`; per-chat extra examples + candidate → `Cacheable: false`.

**Gemini Cache Key**: SHA-256 of (model + system instruction + cacheable prefix), first 12 hex chars as display name `ngbot-spam-<hash>`.

**Gotchas**:
- **Empty-cached-response retry** (commit `4cdc500`): when a cached request succeeds but yields no text, the adapter discards it and re-runs the full prompt uncached — the classifier must always get a real verdict. Cache-use errors (expired/not-found) trigger the same fallback.
- Remote cache listing occurs only on first use/restart or after local-handle invalidation; steady-state requests use the local fingerprint/expiry map.
- Gemini: roles mapped (`assistant` → `model`), system messages extracted as `SystemInstruction` (must precede all conversation messages or it errors)
- Gemini: all four safety categories `BlockNone`; `splitPromptSegments` caches only the contiguous cacheable prefix
- OpenAI: system messages collapsed + prepended; `Cacheable` field ignored
- Both adapters: `MaxOutputTokens=4`, temperature 0 — a deterministic binary verdict generator, not a text producer

### Telegram Update Processing

```mermaid
sequenceDiagram
    participant TG as Telegram
    participant Poll as Polling (backoff/recovery)
    participant Dispatch as Keyed Dispatcher
    participant UP as UpdateProcessor
    participant Admin as Admin Handler
    participant GK as Gatekeeper
    participant Reactor as Reactor

    TG->>Poll: Long-poll (configurable timeout)
    Poll->>Dispatch: Submit update (blocking backpressure)
    Dispatch->>Dispatch: Key by chat, else user, else global
    Dispatch->>UP: FIFO within key; max 8 active workers
    UP->>UP: Check age (< 5 min)

    alt Update too old
        UP->>UP: Drop update
    else Valid update
        UP->>Admin: Handle()

        alt Admin handled
            Admin-->>UP: proceed=false
        else No admin action
            Admin-->>UP: proceed=true
            UP->>GK: Handle()

            alt Gatekeeper handled
                GK-->>UP: proceed=false
            else No gatekeeper action
                GK-->>UP: proceed=true
                UP->>Reactor: Handle()
                Reactor-->>UP: proceed=true
            end
        end
    end
```

### Gatekeeper Challenge Flow — WebApp Join Request (with fallback)

```mermaid
sequenceDiagram
    participant User
    participant TG as Telegram
    participant GK as Gatekeeper
    participant WebApp as WebApp HTTP server
    participant DB
    participant BanService

    User->>TG: Request to join
    TG->>GK: ChatJoinRequest (query_id)
    GK->>BanService: CheckBan (pre-check)
    GK->>DB: Persist generation (pending, token, QueryID, options)
    GK->>TG: sendChatJoinRequestWebApp (open Mini App)
    GK->>WebApp: server listening on ListenAddr

    alt User opens WebApp within 11s
        User->>WebApp: GET /gatekeeper/join-captcha?token=…
        WebApp->>DB: MarkWebAppChallengeOpened (idempotent)
        User->>WebApp: POST /answer (init-data HMAC)
        WebApp->>WebApp: validate init-data + replay (1h) + binding
        WebApp->>BanService: IsKnownBanned (re-check before approve)
        WebApp->>DB: ClaimForApproval(challenge_id) → approve_query_pending
        alt claim won (rowsAffected==1)
            WebApp->>TG: AnswerJoinRequestQuery(approve)
            WebApp->>DB: status → passed_waiting_member_join
            Note over WebApp,DB: failure stores retry metadata; lifecycle worker resumes
        else claim lost (409)
            WebApp-->>User: 409 (concurrent solve)
        end
    else WebApp never opened (>11s) OR expired pending
        GK->>DB: BeginDMFallback(challenge_id), expected pending
        alt claim won
            GK->>DB: status → web_app_fallback_pending
            GK->>GK: prepare/send/bind DM challenge on the same generation
            Note over GK: later chat_member join → greeting + delete row
        else claim lost
            Note over GK: HTTP path already approved; back off
        end
    end
```

> **Crash recovery**: all external-action states are selected by the durable worker using `next_attempt_at`. The row survives process death and is removed only after the required approve/unrestrict/ban/decline action succeeds or Telegram reports it already applied. Exhausted attempts remain durable and are logged.

```mermaid
    else GatekeeperEnabled AND GreetingEnabled
        Gatekeeper->>Group: Send greeting text
    end

    User->>Gatekeeper: Click button
    Gatekeeper->>DB: GetChallengeByMessage (status=pending only)

    alt Correct answer
        Gatekeeper->>DB: CAS pending → unrestrict_pending
        Gatekeeper->>Group: Unrestrict user (retryable)
        Gatekeeper->>Group: Delete challenge message
        Gatekeeper->>DB: Delete exact generation after success
    else Wrong answer (max 3 attempts)
        Gatekeeper->>DB: Atomic attempt increment → reject_pending
        Gatekeeper->>Group: Ban/decline (retryable)
        Gatekeeper->>Group: Delete messages
        Gatekeeper->>DB: Delete exact generation after success
    end
```

### Config Validation Rules

| Constraint | Reason |
|------------|--------|
| `PollTimeout > 0` | Long-poll must wait for updates |
| `RequestTimeout > PollTimeout` | HTTP timeout must exceed poll wait |
| `RecoveryWindow > RequestTimeout` | Recovery must span multiple retry cycles |
| `GatekeeperWebApp.PublicURL` absolute HTTPS when public; HTTP only for loopback | Telegram Mini App traffic must be encrypted outside local development |
| `LLM.RequestTimeout > 0` | Classification calls must have a bounded deadline |

### Log Redaction

`config.RegisterSecret(secret)` + `config.Redact(s)` mask registered literals and Telegram token forms. `NbFormatter` converts `error` and `fmt.Stringer` fields to readable text before a final whole-line redaction pass. Startup configuration exposes only `[REDACTED]`/configured state—never a prefix or suffix. Classification logs contain chat/message IDs, length and SHA digest instead of full user content.

## Conventions

### Go Code Style
- Go 1.25+ with modern tooling (`tool` directive for dev tools)
- Self-documenting code (minimal comments); `gofumpt` formatting
- `any` instead of `interface{}`; context as first parameter

### Architecture
- Hexagonal/DDD structure; interface-based DI
- Consumer-local storage ports (`adminStore`, `gatekeeperStore`, `reactorStore`, `spamStore`, `banStore`, `statsStore`)
- One concrete SQLite adapter (no repository-per-table layer); handler chain of responsibility; `lifecycle.Component`

### Database
- SQLite (pure-Go `modernc.org/sqlite`) with embedded migrations (`rubenv/sql-migrate`)
- `db:` struct tags for `sqlx`; one app-level `sync.RWMutex` serializes writes
- Composite primary keys for parallel operations; upsert pattern (`ON CONFLICT … DO UPDATE`)
- Concurrency control via atomic compare-and-set (`UPDATE … WHERE status=…` + `RowsAffected()==1`)

### Testing
- Table-driven tests beside code (`*_test.go`); `t.Parallel()`
- Integration tests with real SQLite (`t.TempDir()`); HTTP test servers for API tests
- AST parsing for static analysis (i18n consistency)

## Gotchas

| # | Issue | Details |
|---|-------|---------|
| 1 | Handler chain order matters | Admin → Gatekeeper → Reactor |
| 2 | Vote polarity inverted across 3 layers | Buttons `spam_vote:{id}:0`=NotSpam / `:1`=Spam; Reactor maps `'0'`→`vote=true`; `db.SpamVote.Vote==true` = NOT spam. Misreading any layer flips the outcome |
| 3 | Tie votes resolve as false-positive | Timeout + tie → false-positive (user-friendly) |
| 4 | Update timeout is 5 minutes | Old updates silently dropped |
| 5 | Cyrillic normalization is lossy | LLM sees normalized text, not original |
| 6 | External banlist | A failed provider retains its prior snapshot and retry timestamp; effective membership is the union of unexpired provider rows |
| 7 | Gatekeeper fallback | `lang → en → built-in defaults` |
| 8 | Admin panel session TTL | 1 hour, background cleanup every 5 min |
| 9 | CAPTCHA max attempts | 3 wrong answers = temp ban |
| 10 | Reaction profile check | LLM checks unknown reactors (users **and** anonymous channels) before their first chat message; gated by `ReactionProfileCheckEnabled` (admin-panel toggle); removals ignored |
| 11 | Polling recovery window | Default 10m; exceeded → `PollingRecoveryError` → shutdown (exit 1) |
| 12 | Malformed update filtering | `isStructurallyEmptyUpdate` must stay in sync with `api.Update` struct |
| 13 | Polling error channel | The producer emits one terminal error; the channel buffer is intentionally sized for that contract |
| 14 | Command targeting | `commandTargetsCurrentBot` rejects `/voteban@otherbot`; empty bot username rejects all named commands |
| 15 | External quote heuristic | Bypasses LLM entirely — treated as spam immediately |
| 16 | Auto-forward detection | `IsAutomaticForward + IsChannel` → skips entire spam pipeline |
| 17 | Ban durations are temporary | All bans/mutes = 10 minutes (bot issues no permanent bans via this path) |
| 18 | Banlist freshness | Daily snapshots are authoritative only per successfully refreshed provider; hourly rows expire after 26h and positive online checks after 24h |
| 19 | Not-found inconsistency | `ErrNotFound` for settings, `(nil,nil)` for challenges/admin, `("",nil)` for KV |
| 20 | `Settings.Enabled` coupling | Always forced to equal `GatekeeperEnabled` on save |
| 21 | Diagnostic result cache | Capped at 1000 entries, keyed by `{ChatID, MessageID}` and protected by a mutex |
| 22 | Worker group pattern | Uses `wg.Go()` (Go 1.25 `sync.WaitGroup.Go`), not `wg.Add(1)` + `go` |
| 23 | Log formatter depth | `runtime.Caller(6)` assumes specific logrus call stack |
| 24 | i18n key design | Keys ARE English text (not symbolic identifiers) |
| 25 | Callback data size limit | 64 bytes; admin panel sidesteps it via DB-backed `AdminPanelCommand` rows |
| 26 | Not-spammer override scoping | Global overrides (`chat_id IS NULL` or `0`) can only be inserted directly into DB, not via admin panel |

| 28 | Not-spammer bypasses ban check | Override checked *before* ban check — overridden users skip ban DB lookup entirely |
| 29 | Override idempotency | Creating same override twice returns existing row (no error, no duplicate) |
| 30 | Negative chat IDs | Telegram supergroup/channel IDs are negative; override system explicitly supports them |
| 31 | i18n key count | ~135 top-level keys; consistency test checks admin panel keys |
| 32 | Known non-member upsert mutates input | `UpsertChatKnownNonMember` modifies `CreatedAt`/`UpdatedAt` on the passed struct in-place |
| 33 | Known non-member not-found | `getChatKnownNonMember` returns `(nil, nil)` on no rows |
| 34 | Known non-member is optimization, not gate | Remembered users still get re-verified via `GetChatMember` on each message; it's a short-circuit, not a bypass |
| 35 | Challenge status invisibility | `GetChallengeByMessage` filters `status=pending` only; handed-off challenges become invisible to callback handler |
| 36 | Join-request handoff TTL | `approvedJoinRequestChallengeTTL = 5min`; user must appear in group within this window after captcha |
| 37 | Join-request expiry = no penalty | Expired challenges with `CommChatID != ChatID` are cleaned without ban (user changed mind) |
| 38 | Gemini cache contiguous prefix | `splitPromptSegments` only caches until first non-cacheable message; everything after is live |
| 39 | Gemini cache fallback | On cache API failure **or empty cached response**, retries with full uncached content |
| 40 | Gemini cache fingerprint | SHA-256 of (model + system + cacheable prefix); changing any prefix message invalidates cache |
| 41 | Gemini cache TTL | 6 hours; stale caches are replaced on next request |
| 42 | OpenAI ignores Cacheable | `Cacheable` field on messages is a no-op for OpenAI adapter |
| 43 | Stats key pattern | `stats:<chatID>:<YYYY-MM-DD>:<metric>`; stats accumulate indefinitely until KV store cleanup |
| 44 | Recommended protection one-time visibility | Shows only on first `/settings` per chat (pristine defaults); hidden permanently via `admin_panel:recommended_protection_handled:<chatID>` KV flag |
| 45 | `saveChatSettings` side effect | Every settings save via admin panel also writes the recommended-protection-handled KV flag |
| 46 | Challenge status auto-set | `CreateChallenge` auto-sets `status=pending` if empty; callers need not set it |
| 47 | Greeting `mdv2:` prefix stored in DB | `GatekeeperGreetingText` column contains the `mdv2:` prefix; consumers must strip before sending to Telegram API |
| 48 | MarkdownV2 parse-mode upgrade is whole-message | `mdv2:` greeting upgrades the entire combined message (greeting + challenge) to MarkdownV2 |
| 49 | Greeting placeholder restrictions | Placeholders inside `code`/`pre`/`text_link`/`text_mention`/`url`/`mention` entities are rejected — only formatting entities may wrap placeholders |
| 50 | Greeting entity normalization uses UTF-16 offsets | Telegram uses UTF-16 for entity positions; `buildGreetingEntitySpans` converts to rune indices |
| 51 | Empty greeting treated as legacy Markdown | Empty `GatekeeperGreetingText` fails the `mdv2:` check → `greetingParseMode()` falls back to `ModeMarkdown` |
| 52 | Vote-ban report flow | `/voteban` and reply `@bot` re-run a report-specific LLM check; report artifacts are a durable delete-after-10m queue independent of case resolution |
| 53 | CAS provider | Second banlist provider (Combot Anti-Spam) with real-time check API + CSV export; races concurrently with lols.bot |
| 54 | Ban service concurrent checking | `CheckBan` races all providers concurrently; first positive wins; remaining cancelled via child context |
| 55 | Provider fail-open | All provider errors are treated as "not banned" — resilience over false positives |
| 56 | Ban service daily+hourly refresh | Daily replaces only a successful provider snapshot; hourly entries live 26h; bootstrap synchronously loads the effective local snapshot |
| 57 | Ban notification quote fallback | `sendNotificationWithQuoteFallback` retries without quote if Telegram rejects the quote parameter |
| 58 | WebApp server starts conditionally | Only started when `GatekeeperWebApp.PublicURL` is non-empty; the listen addr may be set but the server stays down without a public URL. The server has **no dedicated lifecycle component** — it is owned by `Gatekeeper` (started in `Gatekeeper.Start`, stopped in `Gatekeeper.Stop`) |
| 59 | WebApp atomic approval claim | `ClaimForApproval(challengeID)` transitions `pending → approve_query_pending`; a failure stays durable for retry and never resurrects `pending` |
| 60 | WebApp atomic fallback claim | `BeginDMFallback(challengeID)` wins only for the matching generation in `pending`; a concurrent correct answer prevents fallback |
| 61 | WebApp init-data replay window | `auth_date` validated against `joinCaptchaInitDataTTL = 1h`; `UserID`/`QueryID` bound to the challenge (N1 replay fix) |
| 62 | WebApp known-banned re-check | `IsKnownBanned(userID)` is re-checked **after** a correct answer but **before** claiming — a banned user is declined even with the right choice (N2 fix) |
| 63 | WebApp fast fallback timing | Unopened WebApp challenges fall back to DM captcha ~14s after creation (`webAppOpenDeadline` 11s + 3s sweep), worst case — was ~3min before hardening |
| 64 | WebApp crash recovery | The lifecycle worker resumes fallback/approve/reject states using retry metadata; rows remain after exhausted retries for operator intervention |
| 65 | Gemini empty-cached-response retry | A cached request that succeeds but yields no text is discarded and re-run uncached — the classifier always gets a real verdict (`4cdc500`) |
| 66 | Admin messages protected | Anonymous-admin messages (`msg.SenderChat != nil`) in a group `/settings` invocation are deleted and ignored; prevents spoofed settings triggers (`974d5f2`) |
| 67 | WebApp HTTP server timeouts | `ReadHeaderTimeout` 5s, `Read`/`Write` 15s, `Idle` 60s — slowloris hardening (N10); body capped at 16 KiB |
| 68 | WebApp listen addr must not change to loopback | In Docker the server binds `0.0.0.0:8080` (container) mapped to `127.0.0.1:18080` (host); loopback inside the container is unreachable from the host-side publish proxy. Caddy is the only public TLS entry (N7 deliberately deferred) |
| 69 | Voteban report retention | Rows become due at 10m; Telegram deletion happens first, then the exact artifact row is removed |
| 70 | LLM is a verdict generator | Both providers use temperature 0 + `MaxOutputTokens=4` — deterministic binary output, not prose |
| 71 | Secret redaction thresholds | Token regexes require secret ≥30 chars and numeric id ≥6 digits; shorter `id:value` pairs are only masked via explicit `RegisterSecret` |
| 72 | Explicit configuration | `config.Load()` has no global singleton; runtime constructors receive the validated config value from `cmd/ngbot` |
| 73 | Dependency boundaries | `bot.Service` is settings/membership/language only; handlers receive Telegram and consumer-owned store ports directly |
| 74 | Disabling LLM check does not disable gating | With `LLMFirstMessageEnabled=false` the pipeline returns early into `rememberAuthorIfPossible` (`reactor_message_pipeline.go:149-154`), which still promotes the non-member to member/known-non-member on first sight |
| 75 | `@bot` mention short-circuits the normal pipeline | A reply mentioning the bot routes to `voteBanCommand` (`reactor.go:134-141`) and returns before `handleMessage` — it runs a report-specific `IsReportedSpam` check, NOT the standard `IsSpam` pipeline |
| 76 | `UnmuteUser` restores ALL permissions | Unmute sends `unrestrictedChatPermissions()` (`ban_service_actions.go:50,124-141`) — the full set incl. CanPinMessages/CanManageTopics/CanInviteUsers; no prior-permission snapshot is stored |
| 77 | Gemini cache-error detection is fragile | `isCacheUseError` (`gemini.go:306-313`) is a lowercase substring match (`"cached content"`, `"not found"`, `"expired"`, …) — broad substrings can match unrelated errors |
| 78 | `service.Shutdown()` nils the caches | After shutdown `memberCache`/`settingsCache` are nil (`service.go:352-353`); a subsequent cache-miss **write-back** panics on nil-map assignment (reads are safe) |
| 79 | Handler chain is fail-fast | `UpdateProcessor.Process` returns on the first handler error and short-circuits on `proceed=false`; `selectUpdateHandlers` preserves configured `admin → gatekeeper → reactor` order |
| 80 | Dispatcher key fallback | Chat ID is preferred; updates without a chat use user ID, then one global key. FIFO is guaranteed only within the selected key |
| 81 | Voting authority | Log-channel membership is presentation-only; every vote rechecks source-chat member/admin status |
| 82 | Diagnostic authorization | `/testspam` and `/skipreason` allow `DebugUserID`; group usage also allows source-chat admins, while private usage is debug-user only |
| 83 | Foreign-key enforcement | `_pragma=foreign_keys(1)` is part of the DSN and startup validates it; a connection opened outside `NewSQLiteClient` must set the pragma itself |

## Navigation Guide

### To add a new handler
1. Create package in `internal/handlers/<name>/`
2. Implement `bot.Handler` interface
3. Add compile-time check in `internal/handlers/dependencies.go`
4. Construct it in `cmd/ngbot/buildRuntime` and add it to the `selectUpdateHandlers` availability map
5. Add to `NG_HANDLERS` env var

1. Create package in `internal/adapters/llm/<provider>/`

## Recent Changes (since 2026-05-08)

### Durable Gatekeeper State Machine
- **Files**: `gatekeeper_challenge_service.go`, `gatekeeper_scheduler.go`, `gatekeeper_join_processor.go`, `gatekeeper_webapp.go`, `client_challenges.go`; migration `20260714010000-add-durable-gatekeeper-state.sql`.
- **Change**: every generation receives a unique `challenge_id`; wrong attempts and all lifecycle transitions are expected-state CAS operations. Telegram approve/unrestrict/ban/decline effects run from durable `*_pending` states with bounded exponential retry and restart recovery. Activation failures compensate restrictions and visible messages.
- **Compatibility**: the embedded WebApp, DM fallback, callback payloads and public UI remain unchanged. The internal listener still uses HTTP behind a required TLS proxy; public URLs require HTTPS except loopback development.

### Voteban Report Retention
- **Files**: `spam_control.go`, `client_spam.go`; migrations `20260513` and `2026071402`.
- **Change**: voting resolution and report cleanup are durable. A report artifact becomes eligible after 10 minutes; Telegram deletion precedes deletion of that exact DB row. Case resolution no longer destroys cleanup artifacts early.

### Gemini Empty-Cache Retry
- **Files**: `gemini.go`
- **Change** (`4cdc500`): when a cached Gemini request succeeds but returns empty text, the adapter discards it and re-runs the full prompt uncached. An empty cached response is treated as a stale/invalid cache hit, not a legitimate "no output" — the spam classifier always gets a real verdict.

### Admin Message Protection
- **Files**: `panel_handler.go`, `admin.go`
- **Change** (`974d5f2`): anonymous-admin messages (`SenderChat != nil`) in a group `/settings` invocation are deleted and ignored, preventing spoofed settings triggers. Manager access is re-verified on every callback and free-text input.

### Reaction-Profile-Check i18n
- **Files**: `panel_render.go`, `resources/i18n/translations.yml`
- **Change** (`829e726`): the reaction-profile-check admin description translated into all locales; admin-panel leaf screen for the toggle.

### Localized DM Messages by Client Language
- **Files**: `gatekeeper_join_processor.go`, `gatekeeper_webapp.go`; migration `20260615` (`user_language`)
- **Change** (`a2a720c`): gatekeeper DM messages are localized by the joiner's client language (`user_language` captured on the challenge).

### Earlier entries (still relevant)

#### Multi-Provider Ban Checking (CAS + lols.bot)
- Provider checks still race concurrently, while persistence now tracks provider/feed provenance. Successful daily snapshots replace only their provider, failed providers retain prior data, hourly entries expire after 26 hours, positive online checks after 24 hours, and the legacy `banlist` table is rebuilt as the effective union.

- `/voteban` and reply `@bot` report missed spam; report-specific LLM check first; confirmed spam = instant ban; non-restrict reporters start a message-bound community vote. (See retention change above.)
