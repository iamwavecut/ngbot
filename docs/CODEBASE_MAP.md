## System Overview

**ngbot** is a Telegram chat gatekeeper bot that protects groups from spam and automated bot joins. It provides CAPTCHA-based entry verification with configurable options, LLM-powered spam detection with Gemini explicit caching, external-quote heuristic fast-path, community voting moderation, MarkdownV2 greeting templates with entity-based normalization, and daily statistics tracking.

### Core Bot Module (`internal/bot/`)

**Purpose**: Central bot service, custom polling, and update processing pipeline

| File | Purpose | Key Exports |
|------|---------|-------------|
| `service.go` | Membership/settings management with TTL caching | `NewService()`, `IsMember()`, `GetSettings()`, worker group for background tasks |
| `service_test.go` | Service initialization and lifecycle tests | Integration tests with real SQLite |
| `update_processor.go` | Update dispatch to handlers (chain of responsibility) | `UpdateProcessor`, `RegisterUpdateHandler()`, 19 `MessageType` constants, `StageOverrideCheck`, `StageKnownNonMember` |
| `polling.go` | Custom long-polling with exponential backoff and recovery window | `GetUpdatesChans()`, `PollingOptions`, `PollingRecoveryError` |
| `polling_test.go` | Polling recovery, backoff, malformed update tests | Tests for all polling failure modes |
| `dependencies.go` | Interface contracts | `Service`, `Handler`, `Client` interfaces |

**Key Patterns**:
- Two-level caching: members (5-min TTL with lazy eviction), settings (process-lifetime)
- Handler registry with chain of responsibility (`proceed bool` stops chain)
- Update timeout: 5 minutes (stale updates dropped)
- Custom polling replaces library polling: exponential backoff (initial 1s, max 30s), recovery window (default 10m), malformed update filtering
- Worker group (`sync.WaitGroup.Go()`) replaces raw goroutines

**Dependencies**: `telegram-bot-api`, `db`, `config`, `errgroup`

**Recommended Protection Preset**:
Applies these values in one click:
- Gatekeeper ON, CAPTCHA ON, Greeting OFF
- Challenge timeout: 3m, Reject timeout: 10m
- LLM first-message check: ON
- Community voting: OFF
- Button appears only once per chat (first `/settings` with pristine defaults), then permanently hidden via KV flag
- `saveChatSettings` wrapper marks `admin_panel:recommended_protection_handled:<chatID>` in KV store on every settings save
- `hasCustomizedSettings` detects any deviation from defaults across 13 fields

**Daily Stats Display** (Home Page):
- Aggregates 7 metrics over last 7 days from KV store
- Key format: `stats:<chatID>:<YYYY-MM-DD>:<metric>`
- Four i18n-formatted lines: header, challenges, spam checks, outcomes

**Admin Panel Session Lifecycle**:
1. User runs `/settings` in group → deep-link button → private chat
2. Session created with JSON state, 1-hour TTL
3. Each callback → `findPanelSessionCommand` (brute-force split resolution) → `applyPanelCommand` → re-render
4. Background goroutine cleans expired sessions every 5 minutes
5. Every render creates O(N) command records, old ones deleted at re-render start
6. `saveChatSettings` called on every settings mutation → also marks recommended protection as handled in KV

**Challenge Status Lifecycle**:

```mermaid
stateDiagram-v2
    [*] --> pending: startChallenge (public or join-request)

    pending --> [deleted]: completeChallenge (public group: CommChatID == ChatID)
    pending --> passed_waiting_member_join: completeChallenge (join-request: CommChatID != ChatID)
    pending --> [deleted]: failChallenge (max attempts)
    pending --> [deleted]: processExpiredChallenges (public: penalty ban)

    passed_waiting_member_join --> [deleted]: handleNewChatMembersV2 (user appears in group + greeting)
    passed_waiting_member_join --> [deleted]: processExpiredChallenges (no penalty)

    note right of passed_waiting_member_join
        Extended TTL: 5 minutes (approvedJoinRequestChallengeTTL)
        Invisible to GetChallengeByMessage (filters status=pending)
        Visible to GetChallengeByChatUser (no status filter)
    end note
```

**Status Constants**:
- `ChallengeStatusPending = "pending"` — Initial state, visible to callback handler
- `ChallengeStatusPassedWaitingMemberJoin = "passed_waiting_member_join"` — Handoff state for join-request flow

**Gatekeeper Constants**:
- `captchaSize = 5`, `maxChallengeAttempts = 3`
- `approvedJoinRequestChallengeTTL = 5 * time.Minute`
- Allowed options counts: `{3, 4, 5, 6, 8, 10}`
- Process intervals: new joiners = 1m, expired challenges = 1m
- Greeting placeholders: `{user}`, `{chat_title}`, `{chat_link_titled}`, `{timeout}`
- Greeting template prefix: `mdv2:` tags MarkdownV2-formatted templates; absence = legacy Markdown
- MarkdownV2 greeting causes parse-mode upgrade for the entire combined message (greeting + challenge)

**Community Voting Policy Resolution**:

| Step | Description |
|------|-------------|
| 1. Base policy | From `config.SpamControl` global config |
| 2. Per-chat overrides | From `db.Settings` (if not `SettingsOverrideInherit` = -1) |
| 3. Normalization | Timeout min 5m, MinVoters min 1, MaxVoters min 0, MinVoters% min 0 |
| 4. Required voters | `max(MinVoters, len(members) * MinVoters% / 100)`, capped by MaxVoters |

**Voting Resolution Logic**:

| Scenario | Result |
|----------|--------|
| Spam majority | `spam` status → user banned + `StatSpamConfirmed` |
| Not-spam majority | `false_positive` status → user unmuted + `StatFalsePositive` |
| Tie + timeout | `false_positive` (user-friendly) |
| Insufficient votes + timeout | `false_positive` (user-friendly) |

### LLM Adapters (`internal/adapters/llm/`)

**Purpose**: Abstraction over OpenAI and Google Gemini APIs

```go
type LLM interface {
    ChatCompletion(ctx, messages) (Response, error)
}
```

| Adapter | Default Model | Key Features |
|---------|---------------|--------------|
| `gemini.API` | `gemini-2.5-flash-lite` | Explicit context caching (6h TTL), SHA-256 fingerprinting, `BLOCK_NONE` safety |
| `openai.API` | `gpt-4o-mini` | Supports custom base URL; cache-unaware (`Cacheable` field ignored) |

**Shared Config**: Temperature 0.9, TopP 0.9, MaxTokens 8192 (both adapters)

**Gemini Explicit Caching Architecture**:

```mermaid
flowchart TD
    A[ChatCompletion called] --> B[splitPromptSegments]
    B --> C{Has cacheable prefix?}
    C -->|No| D[Full uncached request]
    C -->|Yes| E[loadOrCreateCache]
    E --> F{Existing cache match?}
    F -->|Yes| G[Use cached prefix + live tail]
    F -->|No| H[Create new cache with 6h TTL]
    H --> G
    G --> I{API call succeeds?}
    I -->|Yes| J[Return response]
    I -->|No| K[Fallback: full uncached request with retry]
    K --> J
```

**Cacheable Message Contract** (`ChatCompletionMessage.Cacheable bool`):
- System prompt: `Cacheable: true` (stable across requests)
- Built-in few-shot examples (~40): `Cacheable: true` (rarely changes)
- Per-chat extra examples: `Cacheable: false` (varies per chat)
- Candidate message: `Cacheable: false` (unique per request)

**Gemini Cache Key**: SHA-256 of (model + system instruction + cacheable prefix), first 12 hex chars as display name prefix `ngbot-spam-<hash>`

**Gotchas**:
- Gemini: roles mapped (`assistant` → `model`), system messages extracted as `SystemInstruction`
- Gemini: all four safety categories set to `BlockNone`
- Gemini: `splitPromptSegments` only caches contiguous cacheable prefix — once a non-cacheable message appears, everything after is "live"
- Gemini: fallback on cache failure retries with full content including system instruction
- OpenAI: system messages reordered to front of slice; `Cacheable` field ignored
- Both adapters discard the `logger` parameter (`_ = logger`)

## Gotchas

| # | Issue | Details |
|---|-------|---------|
| 1 | Handler chain order matters | Admin → Gatekeeper → Reactor |
| 2 | SpamVote.Vote is inverted | `true` = NOT spam, `false` = IS spam |
| 3 | Tie votes resolve as false-positive | Timeout + tie → false-positive (user-friendly) |
| 4 | Update timeout is 5 minutes | Old updates silently dropped |
| 5 | Cyrillic normalization is lossy | LLM sees normalized text, not original |
| 6 | External banlist | Uses lols.bot API with retry/timeout |
| 7 | Gatekeeper fallback | `lang → en → built-in defaults` |
| 8 | Admin panel session TTL | 1 hour, background cleanup every 5 min |
| 9 | CAPTCHA max attempts | 3 wrong answers = temp ban |
| 10 | Reaction threshold | 5 flagged emojis = auto-ban (hardcoded) |
| 11 | Polling recovery window | Default 10m; exceeded → `PollingRecoveryError` → shutdown |
| 12 | Malformed update filtering | `isStructurallyEmptyUpdate` must stay in sync with `api.Update` struct |
| 13 | Polling error channel | Buffer size 1; second error silently dropped |
| 14 | Command targeting | `commandTargetsCurrentBot` rejects `/ban@otherbot`; empty bot username rejects all named commands |
| 15 | External quote heuristic | Bypasses LLM entirely — treated as spam immediately |
| 16 | Auto-forward detection | `IsAutomaticForward + IsChannel` → skips entire spam pipeline |
| 17 | Ban durations are temporary | All bans/mutes = 10 minutes |
| 18 | `knownBanned` map replaced | Each daily/hourly fetch replaces the entire map |
| 19 | Not-found inconsistency | `ErrNotFound` for settings, `(nil,nil)` for challenges/admin, `("",nil)` for KV |
| 20 | `Settings.Enabled` coupling | Always forced to equal `GatekeeperEnabled` on save |
| 21 | `lastResults` LRU | Capped at 1000 entries, keyed by message ID (scoped per-chat by Telegram) |
| 22 | Worker group pattern | Uses `wg.Go()` (custom WaitGroup extension), not standard `wg.Add(1)` + `go` |
| 23 | Log formatter depth | `runtime.Caller(6)` assumes specific logrus call stack |
| 24 | i18n key design | Keys ARE English text (not symbolic identifiers) |
| 25 | Callback data size limit | 64 bytes; `encodeUint64Min` strips leading zero bytes |
| 26 | Not-spammer override scoping | Global overrides (`chat_id IS NULL` or `0`) can only be inserted directly into DB, not via admin panel |
| 27 | Not-spammer rehabilitation | `rehabilitateNotSpammerUser` silently swallows all errors (fire-and-forget) |
| 28 | Not-spammer bypasses ban check | Override checked *before* ban check — overridden users skip ban DB lookup entirely |
| 29 | Override idempotency | Creating same override twice returns existing row (no error, no duplicate) |
| 30 | Negative chat IDs | Telegram supergroup/channel IDs are negative; override system explicitly supports them |
| 31 | i18n key count | 108+ top-level keys; consistency test checks admin panel keys |
| 32 | Known non-member upsert mutates input | `UpsertChatKnownNonMember` modifies `CreatedAt`/`UpdatedAt` on the passed struct in-place |
| 33 | Known non-member not-found | `getChatKnownNonMember` returns `(nil, nil)` on no rows — consistent with challenges/admin panel convention |
| 34 | Known non-member is optimization, not gate | Remembered users still get re-verified via `GetChatMember` on each message; it's a short-circuit, not a bypass |
| 35 | Challenge status invisibility | `GetChallengeByMessage` filters `status=pending` only; handed-off challenges become invisible to callback handler |
| 36 | Join-request handoff TTL | `approvedJoinRequestChallengeTTL = 5min`; user must appear in group within this window after DM captcha |
| 37 | Join-request expiry = no penalty | Expired challenges with `CommChatID != ChatID` are cleaned without ban (user changed mind) |
| 38 | Gemini cache contiguous prefix | `splitPromptSegments` only caches until first non-cacheable message; everything after is live |
| 39 | Gemini cache fallback | On cache API failure, retries with full uncached content (includes system instruction in live content) |
| 40 | Gemini cache fingerprint | SHA-256 of (model + system + cacheable prefix); changing any prefix message invalidates cache |
| 41 | Gemini cache TTL | 6 hours; stale caches are replaced on next request |
| 42 | OpenAI ignores Cacheable | `Cacheable` field on messages is a no-op for OpenAI adapter |
| 43 | Stats key pattern | `stats:<chatID>:<YYYY-MM-DD>:<metric>`; stats accumulate indefinitely until KV store cleanup |
| 44 | Recommended protection one-time visibility | Button shows only on first `/settings` per chat (pristine defaults); hidden permanently via `admin_panel:recommended_protection_handled:<chatID>` KV flag |
| 45 | `saveChatSettings` side effect | Every settings save via admin panel also writes the recommended-protection-handled KV flag — not just the recommended protection action itself |
| 46 | Challenge status auto-set | `CreateChallenge` auto-sets `status=pending` if empty; callers need not set it |
| 47 | Greeting `mdv2:` prefix stored in DB | `GatekeeperGreetingText` column contains the `mdv2:` prefix for MarkdownV2 templates; raw SQL/debugging will show it — consumers must strip before sending to Telegram API |
| 48 | MarkdownV2 parse-mode upgrade is whole-message | When greeting template has `mdv2:` prefix, the entire combined message (greeting + challenge text) is sent as MarkdownV2; challenge text is re-rendered via `renderChallengeTextMarkdownV2` |
| 49 | Greeting placeholder restrictions | Placeholders inside `code`, `pre`, `text_link`, `text_mention`, `url`, `mention` entities are rejected during normalization — only formatting entities (bold, italic, underline, strikethrough, spoiler) can wrap placeholders |
| 50 | Greeting entity normalization uses UTF-16 offsets | Telegram uses UTF-16 for entity positions; `buildGreetingEntitySpans` converts to rune indices — any change to offset handling must account for multi-byte characters |
| 51 | Empty greeting treated as legacy Markdown | Empty `GatekeeperGreetingText` fails the `mdv2:` prefix check, so `greetingParseMode()` falls back to `ModeMarkdown` — safe because gate checks for empty template and returns early |

1. Create package in `internal/handlers/<name>/`

3. Add compile-time check in `internal/handlers/dependencies.go`
